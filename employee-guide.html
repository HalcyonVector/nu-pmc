First Login — Mandatory for Every New User
!
Your account has a temporary password — you must change it before anything else works
The system blocks all features until password is changed
▾
  • Log in with the username and temporary password given to you by IT admin.
  • The system immediately locks every screen. Every API call returns error code MUST_CHANGE_PASSWORD. The only three actions available are: change password, view your own profile, or log out.
  • On the change-password screen, enter your current temporary password, then your chosen new password twice.
  • Password rules (all must pass): Minimum 8 characters, maximum 128 · At least one UPPERCASE letter · At least one lowercase letter · At least one digit · Not on the common-passwords blocklist (which includes welcome@123, nupmc@2026, nuassociates, month+year combos like january2026, city names like bangalore1, and ~50 others) · Must not contain your own username if your username is 4 or more characters long.
  • Once accepted, you are logged in and can access all features assigned to your role.
Forgot password / locked out? There is no self-service password reset — the WhatsApp OTP flow was removed in v3.1 (April 2026). Ask your direct manager or a Principal / Design Principal to reset your password via User Management → your name → Reset Password. They will set a temporary password; you must change it on your next login.

Phase 1: initialising

Principal creates the project. Status = initialising. No site operations, no payments, no drawings can be approved yet. The team works through the 7-item Readiness Gate during this phase.

Phase 2: Readiness Gate — all 7 must be complete to activate

Phase 3: active

All features unlock: daily reports, payments, drawings, GRNs, issues, MOMs, site visits, etc.

Phase 4: completed / on_hold / cancelled

Principal moves the project after handover is done. completed = project delivered and closed. on_hold = temporarily suspended. cancelled = abandoned. These transitions are guarded and audit-logged.

Why you sometimes see "Stale Version" errors — and how the notification system actually works
▾

Stale Version (409) errors

Every record has a hidden version counter. When you open a payment or drawing card, your browser notes the version. When you click Approve, the system checks whether the version still matches. If someone else already approved or edited it, the version has changed → you see a 409 Stale Version error. Fix: refresh the page, then re-check whether action is still needed.

State machine guard

Status changes (e.g. pending_pmc → pmc_approved) are enforced by a database-level WHERE clause: the update only executes if the current status in the database matches what the system expects. Two simultaneous approvals → one wins, one gets "transition failed". Refresh and re-check.

Audit log — permanent and tamper-proof

Every significant action is written to an append-only audit log: who (user ID), what (action), which record (type + ID), when (timestamp), from where (IP address + browser). Nobody can edit or delete audit log entries. If there is ever a dispute about who approved a payment or changed bank details, the audit log is the definitive source of truth.

Notifications — Matrix first, WhatsApp as fallback (v5.23+)

Staff notifications go via Matrix, not raw WhatsApp. Matrix bridges to WhatsApp for vendors/clients without a Matrix room. If a vendor has no Matrix room (Tier B), a manual send task is queued for PMC Head or Finance Admin in the External Comms queue.

Vendor Bank Detail Change — Approve
Dual-approval mandatory — proposer ≠ approver
▾
  • When a vendor's bank/UPI details must change, someone must propose the change and a different person must approve it.
  • Who can propose: Finance Admin or any Head (PMC / Design / Services).
  • Who can approve: If proposed by Finance Admin → Principal or Design Principal must approve. If proposed by a Head → Finance Admin or Principal or Design Principal must approve.
  • Before approving, confirm the vendor was contacted directly (via Matrix or a known phone number) and confirmed the new bank details. Tick the "Vendor confirmed via Matrix" checkbox only after personal verification.
  • Once approved, the bank details are updated and the change is audit-logged with both proposer and approver user IDs.
NEVER approve a bank detail change without personally verifying with the vendor via a trusted channel. Attackers intercept emails and redirect payments. The dual-approval + Matrix verification process exists specifically to prevent this.

Common Mistakes to Avoid
Schedule upload: Always use the Download Template button in the Schedule tab. Column names and order are exact. Wrong column format causes an upload failure. Do not add extra columns.
Vendor 4-step chain: (1) Register → (2) Finance clears → (3) Engage on project → (4) Principal approves. All four must complete before a payment can be raised. If a payment is blocked, check which step is incomplete.
Drawing upload: Drawing number must exist in the register. Revision must be higher than the last uploaded revision for that number. Upload PDF only.
MOM — two separate steps: Approving locks the MOM as an internal record. Issuing sends it to the client. You can approve without issuing. You must type the exact MOM number when issuing — this prevents accidental sends.
Vendor bank change: Dual approval always required. Proposer ≠ approver. Always verify with vendor directly via Matrix or phone before proposing or approving.
Payment amount: If you see a hard block on payment amount, check: (a) is the amount keyed correctly? (b) has the vendor's contract value been updated to reflect any approved CNs? (c) is there already an earlier payment that consumed the remaining balance?
Notifications: Check the bell icon for items waiting on you. The "Needs You" tab shows a count of pending actions. Action items assigned to you show in "My Actions".
Stale Version error: Not a bug. Refresh and re-check. It means someone else acted first.
RFI vs Drawing Query: Drawing Queries (raised via Issues tab with a drawing linked) are informal. Formal RFIs (raised via the RFI tab) are numbered, tracked against a deadline, and generate overdue alerts. Use formal RFIs for any query that needs a documented response on record.
RFI direction matters: Contractor → PMC means the contractor is asking you for a design/spec clarification. PMC → Contractor means you are asking the contractor to clarify something about the site or their scope. Set this correctly when raising — it determines who the expected responder is.

Matrix is our messaging backbone — think of it like a company WhatsApp that we fully control
You receive project alerts, approvals, and vendor updates here
▾

Matrix is an open messaging standard. We run our own Matrix server (hosted by etke.cc) — which means our conversations, files, and alerts stay on our own infrastructure, not on a third-party platform.

The app you use to connect to Matrix is called Element X. It is available free on iOS and Android. Think of Element X the same way you think of WhatsApp — but instead of WhatsApp's servers, it connects to nu Associates' own server.

3
Responding to polls (Approve / Reject buttons)
Used for weekly sign-offs, change notices, and vendor bank changes
▾

When a decision needs your sign-off, the PMC bot sends a poll to the relevant room. A poll looks like a message with selectable options — "Approve" or "Reject" (sometimes with more options for multi-choice decisions).

  • Open Element X → go to the room where the poll appeared.
  • Tap the poll card. You will see the options listed (e.g. Approve / Reject).
  • Tap your choice. The bot registers your vote within seconds and sends a ✅ reaction on your message to confirm it was recorded.
  • Once enough votes are in (quorum is met), the bot closes the poll and the NU PMC app automatically updates the record status.
Polls time out. If a poll stays open without enough votes for too long, it expires. You will get a notification in your DM with the bot telling you the approval expired. You or the proposer must re-raise the request in the app.
Do not vote twice — if you tap a different answer after already voting, your first vote changes. Matrix only counts your most recent vote. Do not accidentally vote both Approve and Reject.

What to do every day — your Matrix routine
Takes 2–3 minutes each morning
▾
  • Morning — check unread rooms. Open Element X. Rooms with bold names have unread messages. Scan through them. Most are automated alerts from the bot — read them so you know what happened overnight or early morning.
  • Respond to polls first. If you see a poll with your name called out or in a room where your vote is expected, respond before checking anything else. Polls time out, app actions don't.
  • Check your bot DM. Your personal DM with the PMC bot contains items directly addressed to you: overdue reports, tasks you are behind on, your own approval expiries.
  • If you see a @mention — someone has explicitly called your attention. The @ badge shows in the room list. These take priority over general room messages.
  • End of day — verify your daily report went in. If you submitted a daily report in the NU PMC app, the bot sends a confirmation to your DM. If you don't see it by 7 PM, check the app and resubmit.
You do not need to reply to most bot messages. They are for your awareness, not for conversation. Only respond when there is a poll or a direct question directed at you.
Single Sign-On (SSO) — One login for everything
🔑
Your nu PMC username and password are all you need — no separate Matrix password
When Element X asks you to sign in, use your normal nu PMC credentials
▾

SSO stands for Single Sign-On. In plain terms: you do not have a separate password for Element X. The same username and password you use for the nu PMC web app is what you use to log into Element X. One set of credentials, everything unlocks.

How to sign in to Element X using SSO

  • Open Element X. When it asks you to sign in, you will see a button that says "Sign in with nu Associates" (instead of a plain username/password box). Tap that button.
  • Element X will open a login page in your phone's browser. This is the standard nu PMC login page — the same one you use for the web app.
  • Enter your nu PMC username and your nu PMC password. Tap Sign In.
  • If your credentials are correct, the browser will automatically hand you back to Element X. You do not need to copy or paste anything.
  • Element X is now signed in. You will see your rooms and start receiving notifications.
Already logged into the web app on the same browser? If you are already signed in to the nu PMC web app on your phone's browser, the login page may approve you automatically without asking for your password again. This is normal — it means the browser recognised your existing session.
Your password never goes to Matrix. When you sign in via SSO, your credentials are entered on the nu PMC server only. Matrix and Element X never see your password — they simply receive a confirmation from our server that says "yes, this person is who they say they are." This is how SSO is designed to work.
IT
IT Admin — when a staff member leaves, one action covers both systems
Deactivate in nu PMC → Element X access ends automatically
▾

Because login is handled by the nu PMC server, deactivating an account there is all that is needed. You do not have to separately delete or disable the person's Matrix account.

  • Go to User Management in the nu PMC app → find the staff member → tap Deactivate Account.
  • The moment the account is deactivated, the server revokes all active SSO tokens. Any device where that person is logged into Element X will be signed out on the next request.
  • The person will not be able to sign back in to Element X because the "Sign in with nu Associates" step will fail — their nu PMC account is inactive.
Do not skip this step when someone leaves. As long as their nu PMC account is active, they can log back into Element X from any device and see project rooms and notifications. Deactivating in nu PMC is the single action that cuts access to everything.

Common problem: "Element X says my credentials are wrong"
Almost always a username issue — not a Matrix issue
▾

If the login page tells you the credentials are incorrect when you try to sign in through Element X, work through these checks before contacting IT Admin:

  • Are you using your nu PMC username? The username is the short one IT Admin gave you (for example ravi.kumar) — not your personal email address, not your @nuassociates.in email. If you are not sure what your username is, check the welcome message IT Admin sent when your account was created, or look at the top of your profile screen in the nu PMC web app.
  • Is your account active? If your account was deactivated (for example during a leave period) it will block login to both the web app and Element X. Try logging into the nu PMC web app first — if that also fails, your account needs to be reactivated. Contact IT Admin or your direct manager.
  • Did you recently change your password? Any device where Element X was already signed in will continue working using the cached session until it expires. After a password change, if you are asked to sign in again you must use the new password on the login page.
If none of the above resolves it, contact IT Admin and tell them: (1) the exact username you are trying to use, (2) whether the web app also fails to log you in or works fine, and (3) the error message you see on the login page. This helps IT Admin diagnose the issue quickly.
For Vendors — You Do Not Need to Install Anything
V
Vendor Tier A vs Tier B — how you receive notifications
Vendors on WhatsApp Bridge (Tier A) get messages directly. Others (Tier B) get a manual send.
▾

Tier A — WhatsApp Bridge (Preferred)

When a vendor is set up on our WhatsApp bridge, they receive messages directly on their normal WhatsApp — no app to install, no account to create. From the vendor's perspective, it looks like a WhatsApp message from a business number. From our side, it comes through our Matrix server. The vendor can reply and the reply appears in the vendor's Matrix room visible to PMC Head and Finance Admin.

  • IT Admin registers the vendor in the NU PMC app (Vendors → master record).
  • The etke.cc WhatsApp bridge creates a bridge room linked to the vendor's WhatsApp number.
  • IT Admin sets the matrix_room_id on the vendor's record via the app (Vendors → vendor → Set Matrix Room).
  • From this point, all automated vendor notifications (payment confirmed, defect raised) go to that room and appear as WhatsApp messages on the vendor's phone.

Tier B — Manual Send (Temporary, until bridge is set up)

If a vendor doesn't have a Matrix room yet, any notification that would go to them is converted into a task in the External Comms queue. PMC Head or Finance Admin sees this task and must manually send the message via WhatsApp (the app provides the wa.me link pre-filled with the message). Once the task is marked done, the notification is logged as sent.

The manual send exists to ensure no notification is silently lost. It is friction by design — the more manual sends PMC Head has to do, the more pressure there is to onboard that vendor to the bridge.

IT Admin — Setting Up Matrix for a New User
IT
Complete checklist: onboarding a new staff member to Matrix
All steps required. Missing any one step = user will not receive notifications.
▾
  • Create the Matrix account at etke.cc admin panel → Users → Create User. Username should match their NU PMC username for easy identification.
  • Set a temporary password and send it to the user via the NU PMC app (Users → their record → Reset Password → Send via Matrix). On first login to Element X, Element X will prompt them to change it.
  • Link the Matrix ID to their NU PMC account → Users → their record → Edit → set Matrix User ID field to @username:matrix.nuassociates.in. Without this, the app doesn't know where to send their notifications.
  • Create their personal DM room with the PMC bot. In Element X (logged in as admin/bot), start a DM with the user → send a welcome message. This room ID is automatically saved to their NU PMC account as matrix_room_id.
  • Add them to their project rooms → for each active project they work on, invite them to the relevant room (Coordination, Finance, etc.) via Element X room settings → Invite.
  • Verify — trigger a test notification from the app (e.g. have them log in and check the bell icon). Confirm they receive it in Element X within 30 seconds.
Step 3 (linking Matrix ID in NU PMC) is the most commonly missed step. The app routes notifications to the Matrix ID stored in the users table. If that field is blank, all notifications for this user silently go to the WhatsApp fallback instead of Matrix.
IT
Setting up a vendor on the WhatsApp bridge
Allows the vendor to receive all future notifications on their normal WhatsApp
▾
  • In the etke.cc dashboard → WhatsApp Bridge section → Contacts → find or add the vendor's WhatsApp number.
  • The bridge will create a Matrix room for this contact. Copy the Matrix Room ID (it starts with ! and contains a colon, e.g. !abc123:matrix.nuassociates.in).
  • In the NU PMC app → Vendors → find the vendor → tap Set Matrix Room (requires IT Admin or Principal role). Paste the Matrix Room ID. Tap Save.
  • The vendor's record now shows Matrix Status: Joined. All future notifications to this vendor route through the bridge automatically.
  • Test it: In the NU PMC app, go to the vendor's project engagement → raise a test defect or send a test notification. Confirm the vendor receives it on WhatsApp within 1 minute.
If the vendor's WhatsApp number changes: Return to etke.cc dashboard → update the contact number → the bridge creates a new room → update the Matrix Room ID in NU PMC to the new room ID. The old room ID will no longer receive messages.
Once the bridge is set, the vendor can also reply via WhatsApp. Their reply appears in the Matrix room visible to PMC Head and Finance Admin. No staff member needs to be awake at 2 AM — the reply is there waiting in the morning.
